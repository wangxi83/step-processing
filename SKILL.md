# 分步处理

## 概述

把一个较大的目标拆成一组可独立执行、可验证、可交接的小步骤。所有步骤文件写入项目根目录下的一个目录，并用 `next-session.md` 作为启动器，指引每个后续 session 从正确位置继续；用 `task-status.md` 记录每轮 session 后的步骤状态、完成证据和阻塞情况，方便新的 session 快速回顾全局进度。

**开始时宣布：** "我正在使用 step-processing 技能来分解任务并生成可接续的 session 文件。"

## 强制规则

- 每个 session 启动或继续既有分步计划时，必须先读取 `task-status.md` 判断整体状态；如果所有步骤均为 `已完成`，必须直接告诉用户“分步骤任务已全部结束”，不得继续执行、重跑或猜测仍有后续工作。
- 必须先向用户确认分解内容要放到项目根目录下的哪个目录中；用户未确认前，不得创建步骤文件。
- 目标目录必须位于当前项目根目录内，不能写到磁盘根目录、用户目录外或归档参考目录。
- 如果目标目录已存在且包含同名文件，必须先读取现有内容并说明覆盖、追加或续写方案；未经用户确认不得覆盖。
- 每个步骤文件必须能被一个独立新 session 直接执行，不依赖当前对话隐含上下文。
- 必须在目录中创建并维护 `task-status.md`，用于记录所有步骤的状态、每轮 session 的处理结果、验证证据和阻塞风险。
- 每次完成或结束一个 session 前，必须同时更新 `next-session.md` 和 `task-status.md`：`next-session.md` 指引下一个 session 从哪里开始，`task-status.md` 记录当前步骤任务情况，便于新 session 回顾任务完成情况。
- 不要把大任务拆成只有标题的空步骤；每个步骤都要有目标、输入、执行清单、验证方式和完成后更新要求。

## 流程

### 步骤 0：继续既有计划时先检查状态

如果用户要求按某个 `<目录>/next-session.md` 继续，或目标目录中已经存在 `task-status.md`：

1. 先读取 `task-status.md`，再读取 `next-session.md`。
2. 检查 `总览` 和 `步骤状态` 表，判断是否仍有 `未开始`、`进行中` 或 `阻塞` 步骤。
3. 如果全部步骤均为 `已完成`，立即停止后续步骤，告诉用户：`分步骤任务已全部结束`，并简要说明最后完成证据或最后更新时间。
4. 如果仍有未完成或阻塞步骤，再按 `next-session.md` 指向的步骤继续。

### 步骤 1：确认目录

先询问用户：

```
分解文件要放在项目根目录下的哪个目录？例如 `final-check/`、`plans/order-refactor/`。
```

确认后：

1. 解析为项目根目录内的相对路径。
2. 检查路径是否已存在。
3. 如存在，读取目录下已有 `.md` 文件，判断是继续已有计划还是新建/覆盖。
4. 如不存在，创建目录。

### 步骤 2：理解并审查目标

读取用户提供的需求、计划或相关文档。必要时读取项目规则文件和涉及模块，但只读取完成分解所需的上下文。

审查重点：

- 目标是否明确。
- 是否存在必须先完成的依赖。
- 是否涉及关键路径，需要 TDD 或更严格验证。
- 是否需要数据库、外部服务、账号、浏览器或环境变量。
- 哪些步骤可以独立 session 执行，哪些必须顺序执行。
- 每一步的完成证据是什么。

如发现目标不清、验收标准缺失或环境依赖无法推断，先向用户澄清。

### 步骤 3：生成步骤文件

在确认目录中生成：

- `next-session.md`：唯一启动器，后续 session 优先读取它。
- `task-status.md`：全局任务状态表，后续 session 用它回顾每个步骤的完成情况。
- `step-01-<short-name>.md`
- `step-02-<short-name>.md`
- 继续按需编号。

文件名要求：

- 使用两位数字编号，保持执行顺序稳定。
- `<short-name>` 使用简短英文 kebab-case，避免空格和中文标点。
- 只生成必要步骤，避免过细导致管理成本过高。

### 步骤 4：步骤文件模板

每个 `step-*.md` 必须包含：

```markdown
# Step NN: <标题>

## 目标

<本步骤要达成的具体结果。>

## 开始前必须读取

- `AGENTS.md`
- `<相关项目规则或上下文文件>`
- `<必要的前置 step 文件>`

## 当前上下文

<足够让独立 session 理解背景的摘要。>

## 执行清单

1. <可执行动作>
2. <可执行动作>
3. <可执行动作>

## 验证方式

- `<具体命令或人工检查点>`
- 预期结果：<明确通过条件>

## 完成后必须更新

- 更新 `next-session.md`，记录下一个 session 入口和本次必要提醒。
- 更新 `task-status.md`，记录本步骤状态、完成证据、验证结果和阻塞情况。
- 如修改代码，记录变更文件和验证命令。
- 如遇阻塞，记录阻塞原因、已尝试内容和下个 session 的建议入口。
```

### 步骤 5：next-session.md 模板

`next-session.md` 必须包含：

```markdown
# Next Session

## 启动指令

按 `<目录>/next-session.md` 开始工作。

## 当前状态

- 计划目录：`<目录>/`
- 当前阶段：<未开始 / 进行中 / 阻塞 / 已完成>
- 下一步：读取并执行 `<目录>/step-NN-<short-name>.md`

## 已完成

- <尚无，或列出已完成步骤和证据>

## 本次应执行

1. 读取 `AGENTS.md` 和必要规则。
2. 读取 `<目录>/task-status.md`，回顾已完成、进行中和阻塞步骤；如果全部步骤已完成，直接告诉用户 `分步骤任务已全部结束`，不要继续执行。
3. 读取 `<目录>/step-NN-<short-name>.md`。
4. 按步骤执行、验证、记录结果。
5. 结束前更新 `next-session.md` 和 `task-status.md`，指向下一个步骤并记录当前步骤任务情况。

## 关键约束

- <项目约束、禁止事项、环境要求>

## 验证记录

- <命令、退出码、摘要、日期时间>

## 阻塞与风险

- <无，或列出具体阻塞>
```

### 步骤 6：task-status.md 模板

`task-status.md` 必须包含：

```markdown
# Task Status

## 总览

- 计划目录：`<目录>/`
- 当前阶段：<未开始 / 进行中 / 阻塞 / 已完成>
- 最后更新：<YYYY-MM-DD HH:mm>
- 当前步骤：`step-NN-<short-name>.md`
- 下一步骤：`step-NN-<short-name>.md` 或 `<无>`

## 步骤状态

| 步骤 | 状态 | 最近处理 session | 完成证据 | 备注 |
| --- | --- | --- | --- | --- |
| `step-01-<short-name>.md` | 未开始 | - | - | - |
| `step-02-<short-name>.md` | 未开始 | - | - | - |

## Session 记录

### <YYYY-MM-DD HH:mm> - <步骤或主题>

- 处理步骤：`step-NN-<short-name>.md`
- 状态变化：<未开始 -> 进行中 / 进行中 -> 已完成 / 进行中 -> 阻塞>
- 完成内容：<具体完成项>
- 验证证据：<命令、退出码、摘要，或人工检查结论>
- 变更文件：<无，或列出路径>
- 阻塞与风险：<无，或具体说明>
- 下次建议入口：`step-NN-<short-name>.md` 或 `<无>`
```

### 步骤 7：session 结束时维护启动器和状态表

任何使用本技能生成的计划，在每次 session 结束前都必须更新 `next-session.md` 和 `task-status.md`。

更新 `next-session.md`：

- 如果当前步骤完成：把 `下一步` 指向下一个未完成步骤。
- 如果全部完成：把 `当前阶段` 标记为 `已完成`，并保留最终验证证据。
- 如果阻塞：把 `当前阶段` 标记为 `阻塞`，写清阻塞条件、已尝试命令、下一次最小可行动作。
- 如果用户改变目标：记录变更内容，并指出哪些步骤需要重写。

更新 `task-status.md`：

- 在 `步骤状态` 表中更新当前步骤状态、最近处理 session、完成证据和备注。
- 在 `Session 记录` 中追加本轮 session 的处理结果。
- 记录验证命令、退出码和关键输出摘要；如未验证，必须说明原因。
- 记录修改过的文件；如没有文件变更，写明 `无`。
- 如果阻塞，记录已尝试动作、失败原因和下次最小可行动作。

## 输出要求

生成或更新文件后，向用户报告：

- 目标目录。
- 生成或更新的文件列表。
- 下一次 session 应使用的启动指令。
- `task-status.md` 中记录的当前步骤状态摘要。
- 任何需要用户确认的风险或前置条件。

## 不做的事

- 不执行步骤本身，除非用户明确要求开始执行。
- 不为未确认的目录创建文件。
- 不把隐含上下文只留在聊天里。
- 不把 `next-session.md` 当作静态说明；它必须随进度持续更新。
- 不把 `task-status.md` 当作可选文件；它是每轮 session 状态回顾的必需文件。
